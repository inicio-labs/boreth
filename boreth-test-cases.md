## LEGEND

- **[CRITICAL]** — Wrong behavior = consensus failure, chain split, or data corruption
- **[HIGH]** — Wrong behavior = node stuck, sync failure, or incorrect state
- **[MEDIUM]** — Wrong behavior = performance degradation, suboptimal behavior
- **[LOW]** — Wrong behavior = cosmetic, logging, or non-consensus issues
- **[EXISTING]** — Test already exists (noted for gap analysis)
- **[MISSING]** — No test exists yet — **action required**

## 1. RECEIPT KEY ENCODING (THE MOST DANGEROUS AREA)

The bor receipt key is `keccak256("matic-bor-receipt-" + blockNumber_BE_u64 + blockHash_raw)`.
Current code in `receipt_key.rs` is **WRONG** — it uses `keccak256(block_hash)` and `keccak256(block_number.to_be_bytes())` instead of the correct composite key with the "matic-bor-receipt-" prefix.

### 1.1 [CRITICAL][MISSING] Receipt key must include "matic-bor-receipt-" prefix
```
The Go implementation uses:
  key = keccak256("matic-bor-receipt-" + number_be_u64 + hash_raw_32)

Current Rust code does:
  legacy: keccak256(block_number.to_be_bytes())     // WRONG - missing prefix
  new:    keccak256(block_hash)                       // WRONG - missing prefix and number

Test: Compute receipt key for known mainnet block and cross-verify with Go output.
Blocks to test: 1_000_000, 38_189_056 (Delhi), 50_523_000 (Agra), 80_084_799 (last pre-Madhugiri), 80_084_800 (first post-Madhugiri)
```

### 1.2 [CRITICAL][MISSING] Block number encoding MUST be big-endian 8-byte
```
Test cases:
  - block 0:           [0,0,0,0,0,0,0,0]
  - block 1:           [0,0,0,0,0,0,0,1]
  - block 256:         [0,0,0,0,0,0,1,0]
  - block 2^32:        [0,0,0,1,0,0,0,0]
  - block 2^32-1:      [0,0,0,0,255,255,255,255]
  - block u64::MAX:    [255,255,255,255,255,255,255,255]

Verify: little-endian encoding produces DIFFERENT key (proves BE is mandatory)
Verify: string encoding "80084800" produces DIFFERENT key (proves binary is mandatory)
Verify: hex encoding "0x04C75F00" produces DIFFERENT key
```

### 1.3 [CRITICAL][MISSING] Block hash must be raw 32 bytes, NOT hex-encoded
```
Test: hash B256::from([0xab; 32])
  Correct: feed raw 32 bytes into hasher
  Wrong:   feed "abababab..." (64 hex chars) into hasher
  Verify they produce different results.
```

### 1.4 [CRITICAL][MISSING] Receipt key transition at Madhugiri boundary
```
Block 80_084_799: uses legacy key format
Block 80_084_800: uses new hash-based key format
Test: Same block hash at both blocks produces DIFFERENT receipt keys
Test: Legacy key at 80_084_799 matches Go Bor output for that block
Test: New key at 80_084_800 matches Go Bor output for that block
```

### 1.5 [HIGH][MISSING] Receipt key collision resistance
```
Test: Generate keys for 10,000 consecutive blocks — all must be unique.
Test: Same block number with different hashes → different keys.
Test: Different block numbers with same hash → different keys (legacy path).
```
